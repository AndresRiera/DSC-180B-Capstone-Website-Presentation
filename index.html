Introduction

The problem with trusting digital files

Every day, contracts are signed, research datasets are shared, and images are used as evidence — yet recipients often have no reliable way to confirm whether a file is genuine, whether it has been altered, or whether it comes from a trusted source. Advances in AI-generated content and editing tools have made convincing forgeries easier than ever to produce.

Traditional solutions rely on centralized institutions that introduce delays, costs, and single points of failure. And while early blockchain systems could prove a file existed at some point in time, they couldn't answer the questions that matter most: Who registered it? Is it still valid? Has it been revoked?

Vera solves this. By recording cryptographic fingerprints — not the files themselves — on the Ethereum blockchain, Vera provides permanent, tamper-proof proof of a file's authenticity, issuer identity, and current status, verifiable by anyone in seconds at nearly no cost.

🔒 The Old Way

Centralized authorities, manual review processes, and platform-specific trust assumptions. Slow, expensive, and vulnerable to fraud or institutional failure.

⛓️ Simple Blockchain (Before Vera)

Existing blockchain proofs only showed a file existed — they couldn't enforce who registered it, confirm its current validity, or support revocation when circumstances change.

✅ Vera's Approach

Cryptographic fingerprints anchored on-chain with full lifecycle management. Authorized issuers register assets. Anyone can verify. Revocation is instantaneous and auditable. Your raw file never touches the blockchain.

How It Works

Verification without a middleman

Vera is built around a simple but powerful insight: you don't need to store a file to prove it's real — you just need to store its fingerprint. A cryptographic hash is a unique, fixed-length string derived mathematically from a file's exact contents. Change even one byte, and the hash changes completely. Vera stores these fingerprints on-chain so anyone can check them independently, forever.

Fig. 1 — Vera's layered architecture: a web interface feeds into application logic (client-side hashing, wallet connection, state queries), which communicates with Ethereum smart contracts, backed by an off-chain database for metadata and governance.

Three roles, one infrastructure

Vera separates everyone involved in verification into three distinct roles, each with a tailored experience built on the same underlying system:

🏛️

Issuers

Authorized organizations — universities, governments, companies — who register, update, or revoke assets. Issuers connect a crypto wallet to sign blockchain transactions.

🙋

Asset Holders

Individuals or organizations assigned a verified asset. They can view their assets and share verification proofs with anyone who needs to confirm them.

🔍

Verifiers

Anyone who wants to check a file — no account, no wallet, no fees required. Upload the file, get an instant on-chain result: Active, Revoked, or Unverified.

Registering an asset

When an issuer registers a new asset, the file never leaves their device. Only its cryptographic fingerprint gets committed to the blockchain:

Fig. 2 — Asset Registration Workflow: Upload → Hash (SHA-256, local) → Wallet signs → Smart contract validates issuer → Store hash + metadata → Emit event.

Technical detail: how SHA-256 fingerprinting works

Every file is read as raw binary data. Vera applies the SHA-256 cryptographic hash function (recommended by NIST) to produce a 256-bit digest that uniquely represents that file's exact contents. This process runs locally on the user's device — nothing is uploaded before the hash is computed.

  • Deterministic: the same file always produces the same hash.
  • Collision-resistant: it is computationally infeasible to find two different files with the same hash.
  • One-way: the original file cannot be reconstructed from its hash, preserving privacy.

Asset-specific metadata is recorded alongside the hash: datasets may include version identifiers; visual media may include format metadata. Smart contract storage maps each hash to a structured record containing the issuer address, registration timestamp, asset type, and a current validity status flag.

Verification is completely open and read-only — no wallet, no account, no gas fees required at any point:

A modular smart contract system

At the core of Vera is a factory-based smart contract architecture. A RegistryFactory contract dynamically spawns separate registries for documents, datasets, and media. This keeps asset categories modular and independently maintainable while sharing the same verification interface and revocation logic.

Fig. 4 — Multi-Registry Architecture. One access control layer feeds into three specialized registries (Document, Dataset, Media), all routing through a shared Revocation Mechanism.

Technical detail: hybrid governance model

Vera uses a deliberate two-layer governance approach that balances security with flexibility:

  • On-chain: Registry contracts enforce an onlyOwner restriction — only the authorized wallet can submit state-changing transactions (registrations, revocations).
  • Off-chain (Supabase RLS): Fine-grained role distinctions — Administrators vs. standard Users — are enforced through database Row-Level Security policies. Admins can register and revoke; standard users can only view and verify. Registry owners can invite additional administrators without redeploying any contracts.

This hybrid model means governance rules can be updated dynamically while immutability guarantees on the blockchain remain intact for all on-chain records.

Assets have a lifecycle, not just a timestamp

Unlike simple proof-of-existence systems, Vera treats verification as a dynamic state. An asset can be Active, Revoked, or Superseded. Status changes are implemented as flag transitions — old records are never deleted, so the full history remains auditable at any time.

Fig. 5 — Asset Lifecycle State Model. Revocation updates a status flag rather than deleting the record, preserving historical auditability.

Results & Conclusion

What we proved it can do

The system was validated under live public test network conditions — not a controlled simulation — across Ethereum Sepolia, Arbitrum Sepolia, and Polygon Amoy. All core functionality was confirmed to work reliably at production scale.

✅ Perfect Registration Accuracy

Across all three asset types, computed hashes consistently matched on-chain records. Identical files always produced identical hashes; any modification produced a completely different fingerprint with no collisions.

🔒 Governance Enforcement Confirmed

Unauthorized users were blocked from restricted operations through coordinated on-chain ownership checks and database policy enforcement. Admins completed full registration and revocation workflows without issue.

🔄 Lifecycle Transitions Work

Assets were registered, updated, assigned to users, and revoked — all while maintaining complete historical auditability. Revoked assets remained queryable but were clearly flagged as invalid.

⚡ Frictionless Third-Party Verification

Verifiers confirmed asset authenticity with zero gas fees and no wallet required. Modified files were correctly rejected; unregistered files returned negative responses; revoked assets were correctly identified.

📱 End-to-End Workflow Validated

All user workflows — authentication via Supabase, wallet connection via WalletConnect, registry creation, asset registration, and verification — executed successfully across all supported test networks.

👤 Usable Without Blockchain Expertise

Test participants completed full issuer and verifier workflows with no prior blockchain experience. Complex cryptographic and transaction-signing operations were abstracted into clear guided steps.

What this means

Vera demonstrates that a decentralized verification system can simultaneously achieve correctness, usability, and governance flexibility — something previous systems have consistently struggled to balance. It goes beyond static proof-of-existence to provide full lifecycle management of heterogeneous digital assets.

By combining on-chain hash anchoring with off-chain storage and governance enforcement, Vera preserves cryptographic trust guarantees while remaining institutionally manageable and accessible to everyday users without blockchain expertise.

Future development will expand contract-level access control, optimize network selection for production environments, and strengthen decentralized governance — moving Vera toward real-world deployment across legal, academic, scientific, and media domains.

Discussion: limitations and future directions

The current implementation has a few known limitations worth acknowledging:

  • Partial off-chain governance: Fine-grained role management currently relies on Supabase Row-Level Security policies. Full decentralization of role management would require additional on-chain access control logic (e.g., OpenZeppelin AccessControl).
  • Gas variability: Write operations (registration, revocation) remain subject to network congestion and gas price fluctuations, though the primary Polygon deployment keeps this minimal (~$0.01/tx).
  • Factory pattern tradeoffs: The RegistryFactory pattern improves scalability but adds an abstraction layer requiring careful version management during contract upgrades.

Planned improvements include contract-level role abstractions, optimized layer-2 deployment strategies, and enhanced cross-chain interoperability to support production-scale institutional deployments.
